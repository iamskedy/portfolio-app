💬 Scalable Real-Time Chat Application

📌 Project Overview

A distributed real-time messaging platform capable of handling 5M+ concurrent users and 120K messages/sec. The system ensures sub-100ms delivery latency using persistent WebSocket connections and an event-driven architecture. The system supports real-time messaging, offline delivery, read receipts, typing indicators, and horizontal scaling across multiple server clusters. The architecture follows design principles used in large-scale messaging platforms such as WhatsApp, Slack, and Discord.

🚨 Problem Solved

Polling-based chat systems introduce significant latency and unnecessary server load. This system replaces HTTP polling with persistent WebSocket connections enabling real-time communication while reducing server overhead by approximately 70%. The architecture is designed to support millions of concurrent connections while ensuring reliable message delivery and fault tolerance.

🏗 High Level Architecture

Users ↓ Global CDN (Cloudflare) ↓ AWS Application Load Balancer ↓ WebSocket Gateway Cluster ↓ Authentication Service ↓ Kafka Message Broker ↓ Chat Service Cluster ↓ Redis Cluster (Presence + Cache) ↓ MongoDB Sharded Cluster ↓ Object Storage (Attachments)

🔄 Message Flow

1. User A sends message via WebSocket 2. Gateway server validates JWT token 3. Message pushed to Kafka topic 4. Chat service consumes message event 5. Message stored in MongoDB 6. Redis checks recipient online status 7. If online → delivered instantly 8. If offline → stored for later delivery 9. Push notification triggered

🗄 Database Schema

Users Collection - userId (indexed) - username - email - profileImage - lastSeen - createdAt Messages Collection - messageId - senderId (indexed) - receiverId (indexed) - roomId - message - attachments - timestamp (indexed) - status ChatRooms Collection - roomId - participants - lastMessage - updatedAt

⚙ Low Level Design

Authentication Service JWT-based login system ensuring secure API access.
Chat Service Handles message creation, delivery, persistence, and retries.
Presence Service Tracks online/offline users using Redis and WebSocket heartbeats.
Media Service Handles file uploads and stores them in AWS S3.
Rate Limiter Prevents spam by limiting users to 20 messages/sec using Redis token buckets.

📈 Scaling Strategy

  • Horizontal scaling of WebSocket servers
  • Redis Pub/Sub for cross-server message delivery
  • Kafka partitions for high throughput streaming
  • Sticky sessions for WebSocket load balancing
  • Auto scaling using Kubernetes HPA
  • Multi-region failover deployment
